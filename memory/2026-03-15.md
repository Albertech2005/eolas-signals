**Fix 1 (commit `602e2ad`):** Split monolithic `EIP712_TYPES` into `REGISTRATION_TYPES`, `ADD_ORDERLY_KEY_TYPES`, `WITHDRAW_TYPES` — ethers v6 `signTypedData` requires a single unambiguous primary type.

**Fix 3 (commit `0c5df56`):** Fixed `getAccountId()` ABI encoding order: `['address', 'bytes32']` (was `['bytes32', 'address']`).

**Fix 4 (commit `c6050ca`) — WRONG, reverted in next commit:**
Attempted to make Safe = Orderly account + delegateSigner approach. This failed because Orderly's `/v1/register_account` endpoint does a direct `ecrecover(sig) == userAddress` check — it doesn't know about `delegateSigner`. So passing `userAddress = safeAddress` with an agent wallet signature gets rejected with "address and signature do not match".

**Fix 5 (commit `cf215b8`) — CORRECT final fix:**
Root issue: Vault's `deposit(data)` sets `receiver = msg.sender`. Since Safe calls Vault via Roles, `msg.sender = Safe address`, so accountId must match Safe — but agent wallet is the Orderly account.

**Solution:** Use `Vault.depositTo(agentWallet.address, data)` instead of `deposit(data)`. `depositTo` sets `receiver` explicitly, so accountId validates against agent address, which matches the Orderly API account. No `delegateSigner` needed.

Changes in cf215b8:
- `VAULT_ABI`: added `depositTo(address receiver, ...)`, removed `delegateSigner`
- `callVaultDeposit`: calls `depositTo(agentWallet.address, depositData)` not `deposit(depositData)`
- `getDepositFee(agentWallet.address, ...)` — receiver must match
- All Orderly API calls (register, addKey, orderlyFetch, buildAuthHeaders) use `agentWallet.address`
- Removed delegateSigner step from setup; restored [1/4] step numbering

### Final architecture
- Agent wallet = Orderly account (registration, API auth, accountId)
- Safe = on-chain source of USDC funds, calls `depositTo(agentAddress, data)` via Roles
- Withdrawals land in agent wallet; `collect` command sends to Safe
- accountId = `keccak256(abi.encode(agentWallet.address, brokerHash))` (encoding order: address first)
